01
Phase 1: Project Initiation & Requirements Gathering
UI · Homepage Input
1. Project Conception Prompt
User enters a project description via the natural language input field.
Input: "Build an automated terrarium controller with a DHT22 temperature/humidity sensor, SG90 servo motor for ventilation, SSD1306 OLED display, and an ESP32 microcontroller."
IN: User Free-Text prompt OUT: POST request body payload { idea: string, model: string }
API · POST
/api/new-flow/start
Instantiates a session doc in MongoDB tracking Q&A logs and confidence matrix. Returns the first clarifying question.
IN: { idea: "Build an automated...", model: "gemini-2.5-flash" } DB WRITE: New NewFlowSession doc OUT: { sessionId, question, options, done: false }
Service · Discovery Agent (Agent 1)
Discovery Agent Initiation
Understands user requirements, compiles system constraints, and decides the next architectural clarification question.
IN: raw prompt text OUT: JSON { question: "...", options: [...], done: false }
Process · Chat Q&A Loop
2. Clarifying Conversation & Data Persistence (3 Faked Q&As)
The user answers questions sequentially via the UI. Each response calls POST /api/new-flow/answer. The backend appends these Q&A pairs directly to the qaHistory array in MongoDB.
[Q&A Round 1] - AI Question: "How would you like the terrarium data to be displayed?" - Options Chips: ["SSD1306 OLED", "LCD1602 Display", "Serial Monitor Only"] - User Input Action: Clicks "SSD1306 OLED" - Payload Sent: { sessionId, answer: "SSD1306 OLED", currentQuestion: "How would you..." }
[Q&A Round 2] - AI Question: "What is your preferred power source for this setup?" - Options Chips: ["USB 5V", "18650 Battery Pack", "9V Wall Adapter"] - User Input Action: Clicks "USB 5V" - Payload Sent: { sessionId, answer: "USB 5V", currentQuestion: "What is your preferred..." }
[Q&A Round 3] - AI Question: "Do you need WiFi telemetry to log data to a server?" - Options Chips: ["Yes (Blynk / MQTT)", "No (Local only)", "BLE Telemetry"] - User Input Action: Clicks "No (Local only)" - Payload Sent: { sessionId, answer: "No (Local only)", currentQuestion: "Do you need WiFi..." }
[MongoDB Persistence - NewFlowSession.qaHistory] [ { "question": "How would you like...", "answer": "SSD1306 OLED", "timestamp": "..." }, { "question": "What is your preferred...", "answer": "USB 5V", "timestamp": "..." }, { "question": "Do you need WiFi...", "answer": "No (Local only)", "timestamp": "..." } ]
DB WRITE: session.qaHistory array updated in MongoDB
UI · Click Action
3. Trigger "Skip Q&A" Option
User wants to bypass further interviews. Clicks "Proceed to Sourcing" in the UI. Fires proceeding API endpoint.
OUT: POST /api/new-flow/proceed
API · POST
/api/new-flow/proceed
Receives session ID. Reads session.qaHistory from the database, converts the array of Q&As into structured context logs, and passes it to the Discovery LLM.
Prompt Constructed for LLM Context: "Original Project Idea: Build an automated terrarium controller... Q&A History: 1. Q: How would you like the terrarium data to be displayed? A: SSD1306 OLED 2. Q: What is your preferred power source for this setup? A: USB 5V 3. Q: Do you need WiFi telemetry to log data to a server? A: No (Local only) The user has decided to skip further questions. Please extract and populate the final context object..."
IN: sessionId OUT: { success: true, context: { ... } }
DB · MongoDB
Finalize Discovery Context
The LLM interprets the Q&A answers and maps them to the structured context document, which is saved to the session. Sets phase1Complete: true.
Saved Context Doc (session.context): { "corePurpose": "Terrarium controller", "mcu": "ESP32 DevKit v1", "subsystems": ["Sensing (DHT22)", "Actuation (SG90 Servo)", "Display (SSD1306)"], "powerSource": "USB 5V 500mA", "connectivity": "None", "constraints": ["Compact enclosure", "USB Powered"] }
DB WRITE: session.context & phase1Complete
↓ phase 1 complete → triggers autonomous formulation
02
Phase 2: Autonomous Formulation Loop (Agent 2)
API · POST
/api/new-flow/formulate
Starts background execution of Agent 2 (Formulation Agent) in an asynchronous multi-turn loop. Returns early confirmation.
IN: { sessionId } OUT: { success: true, message: "Agent 2 formulation initiated" }
Agent Loop · runAgent2()
Agent 2 Task Context & System Instruction
Agent 2 acts as a hardware architect. Operates in a loop (max 20 turns) to search components, verify specs, design wiring, and outline milestones. Communicates intermediate steps back to UI via Socket.IO.
parallel lookup & selection
Tool Call · search_library
Find Subsystem Components
Searches locally stored registry & queries Nexar Graph API for real electronic parts that match constraints.
IN: { query: "ESP32 DevKit v1", category: "MCU" } OUT: List of matching part profiles
Tool Call · get_part_details
Fetch Datasheet & Pin Specs
Loads selected part's detailed schema parameters, mechanical shapes, and pin interfaces.
IN: { partId: "part_esp32_devkit" } OUT: { mpn, pins: [...], specs: { voltage: 3.3 } }
Tool Call · check_compatibility
Verify Electrical Layouts
Ensures peripheral voltage levels and signal protocols match the MCU interfaces (e.g. DHT22 GPIO, SSD1306 I2C).
IN: { partIdA: "ESP32", partIdB: "SSD1306", connectionType: "i2c" } OUT: { compatible: true }
Tool Call · estimate_power_budget
Validate Current Consumption
Aggregates maximum current draw and flags if USB 5V (500mA limit) is overloaded.
IN: { parts: [{ partId: "ESP32", qty: 1 }, { partId: "Servo", qty: 1 }], powerSource: "USB 5V" } OUT: { totalCurrentDraw: 350mA, passes: true }
Tool Call · save_progress
1. Persist BOM (type = "bom")
Saves the compiled list of parts to MongoDB. Emits live BOM update event to client.
IN: data: serialized BOM json DB WRITE: NewFlowSession.bom updated OUT: Socket event 'agent2:bom_update'
Tool Call · generate_wiring
Create Electrical Pin Netlist
Automatically assigns interconnects between components based on functional pin types (SDA, SCL, PWM, Signal, VCC, GND).
IN: { parts: [...], mcu: "ESP32" } OUT: Connections list [{ from: "mcu.D21", to: "display.SDA", color: "#000" }]
Tool Call · validate_pin_assignment
Verify Pin Restrictions
Audits MCU pin usage. Detects pins conflicts, double-allocated pins, or invalid peripheral connections.
IN: { mcu: "ESP32", assignments: [...] } OUT: { ok: true, errors: [] }
Tool Call · save_progress
2. Persist Wiring (type = "wiring")
Saves netlist. Normalizes pins (maps "brain" keyword to "mcu"). Notifies the frontend viewport.
IN: data: serialized connections json DB WRITE: NewFlowSession.wiring updated OUT: Socket event 'agent2:wiring_update'
Tool Call · generate_milestone
Curriculum Step Generation
Creates a step-by-step firmware tutorial. Always sets Milestone 1 to bare blink. Sequential runs cover sensor read, display output, and servo actuation.
IN: Subsystem name & parts involved OUT: Milestone { order, title, code, explanation }
Tool Call · generate_diagram_json
Wokwi Simulator Mapping
Converts aligned connections and part keys into a Wokwi diagram.json schema structure.
IN: { parts, connections } OUT: Wokwi diagram configuration JSON
Tool Call · save_progress
3. Save Curriculum & Diagram
Saves milestones and diagram layouts. Triggers client socket updates.
DB WRITE: session.milestones / session.diagram OUT: 'agent2:milestone_update' & 'agent2:diagram_update'
Tool Call · generate_final_sketch
Integrated sketch.ino Code
Integrates all subsystem code blocks into one complete firmware sketch for local execution.
IN: All milestones, BOM, and wiring netlist DB WRITE: Saves finalSketch to session
↓ formulation completed → client transitions to playground
03
Phase 3: 3D CAD Ingestion Pipeline (Parallel Sourcing)
API · POST
/api/parts/import
Triggered when placing a component onto the playground. Checks local cache first. If not ingested, initiates CAD conversion workers.
IN: { uuid: "easyeda_uuid" } OUT: { glbUrl, pins[] } or { jobId }
parallel fetch & parse
External · EasyEDA 3D Storage
Download CAD File
Fetches raw STEP or WRL footprint file from EasyEDA servers to a local backend temp directory.
OUT: CAD file downloaded (.step / .wrl)
External · EasyEDA API
Download Detail Document
Queries component metadata JSON containing the dataStr.shape[] schema primitives array.
OUT: dataStr.shape[] layout array
Service · model.service.ts
Pin Alignment & Axis Transform
Parses shape[] PAD~ entries to locate pins relative to footprint center.
Transforms scale: mils to mm (× 0.0254).
Flips Y-axis (EasyEDA Y↓ → Three.js Y↑).
IN: shape[] details OUT: pins[] in relative mm coordinates
Service · blender.service.ts
Headless Blender CAD Conversion
Spawns background process:
blender --background --python convert.py
Imports VRML/STEP, shifts mesh to bounding box center, writes bbox center to sidecar _meta.json, and exports binary .glb.
IN: downloaded STEP/WRL file OUT: centered .glb mesh file + _meta.json
Service · model.service.ts
Pin Offset Offset Mapping
Reads bbox center offset from sidecar _meta.json. Subtracts offset from the relative pin layout mm coordinate array, aligning pins exactly with the centered 3D mesh.
IN: pins[], bbox_center OUT: alignedPins[]
DB · MongoDB
Upsert Ingested Part
Saves details to MongoDB Part collection, mapping glbUrl and aligned pins. Flags isIngested: true.
DB WRITE: Part document
↓ assets resolved → loaded in playground
04
Phase 4: Build, Compilation & Live Simulation
Three.js · Simulator3D.tsx
3D Component Render (GenericPart)
Renders 3D components on canvas using useGLTF(glbUrl).
Generates mouse-interactive cylinders for wire nodes using alignedPins[] relative offsets: (pin.x, 0, pin.y).
IN: glbUrl, alignedPins[] OUT: interactive components visible on canvas
UI · CodeEditor.tsx
Firmware Editor
Displays final integrated sketch.ino code. Enables user to make manual edits and trigger firmware validation checks.
IN: sketch.ino code OUT: code updates sent on compile request
API · POST
/api/build/:id/compile
Saves current sketch to a local temporary path. Executes compilation tasks using the local microcontroller CLI tool.
IN: { sketchCode: "void setup()..." } OUT: { success: true, hex, errors: [] }
Service · wokwi-local.service.ts
Headless Arduino-CLI Wrapper
Creates temp build directory. Spawns compiler process:
arduino-cli compile --fqbn esp32:esp32:esp32 --output-dir <tmp>
Returns compilation logs, warnings, or firmware HEX binaries.
IN: sketch.ino file OUT: compiled binary .hex / .bin data
Three.js / WebAssembly · WokwiSimulator.tsx
Live Interactive WebAssembly Simulation
Mounts the client-side Wokwi simulator wrapper.
Loads the compiled hex binary onto the simulated microcontroller.
Renders real-time pin wiring overlay lines. Listens to virtual GPIO signals and outputs console logs to the serial monitor console.
IN: Wokwi diagram.json configuration + compiled HEX binary OUT: LED state toggles, servo motor rotation, sensor serial prints
↓ physical assets packaging
05
Phase 5: Enclosure Packaging & Sourcing Sourcing
API · POST
/api/assembly/:id/layout
Retrieves physical component dimensions from the active BOM. Runs spatial packing calculations to layout cutting blueprints.
IN: BOM component details OUT: SVG layout code, dimensions, component coordinates
Service · assembly.service.ts
2D Bin Packing Layout Engine
Uses heuristic spacing algorithms to place components on an enclosure board with minimal wasted space. Generates SVG cutout vector profiles.
IN: Component sizing mm dimensions OUT: Packed coordinates mapping
UI Render · Forge3D.tsx
3D Enclosure Blueprint
Displays spatial component arrangements. Users adjust coordinates within bounding boxes inside a WebGL viewport.
IN: SVG layout + component coordinates
API · GET
/api/shopping/:id
Retrieves BOM components. Matches items to database price index logs and compiles storefront checklists.
IN: Session BOM OUT: Sourced component checklist + DIY replacement tips
Service · shopping.service.ts
DIY "Jugaad" Sourcing Service
Identifies components with high retail prices and outputs low-cost alternative suggestions (e.g. using hot glue mounting brackets).
IN: Component list OUT: Jugaad swaps mappings
UI · ShoppingPage.tsx
Final Sourced Checklist
Displays unit costs, links to distributors, and final budget totals. Marks project pipeline status as completed.
IN: Sourced checklist details
